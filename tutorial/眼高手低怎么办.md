这种“一看就会，一做就废”的状态（眼高手低），是每个程序员——从新手到大厂架构师——都经历过的必经阶段。

它的本质原因是：**“理解逻辑”属于逻辑记忆（点），而“写出代码”属于工程实践和肌肉记忆（面）。** 你的大脑记住了终点，但你的手指还没学会走那段路。

要解决这个问题，你需要从以下几个实操方法入手，强迫自己把“虚”的理解转化成“实”的能力：

### 1. 放弃“全文复刻”，改用“伪代码中转站”
当你觉得“懂了”的时候，不要直接去写代码。
*   **做法：** 关掉源码，在编辑器里先用**中文注释**把逻辑写下来。
    *   `// 1. 获取用户输入的手机号`
    *   `// 2. 校验手机号格式是否正确`
    *   `// 3. 调用后端发送验证码接口`
    *   `// 4. 根据返回结果显示成功提示或错误信息`
*   **目的：** 如果你连中文逻辑都写不出来，说明你根本没懂；如果你写出了中文逻辑但写不出代码，说明你只是不熟悉 API 或语法。这就把问题拆解了。

### 2. 刻意练习“从零开始”（The Blank Screen Test）
很多新手复刻项目是看着源码改，这太容易了。
*   **做法：** 每天抽出 30 分钟，**不开任何已有代码**，在一个全新的文件夹里，尝试实现原项目中的一个**极小的功能点**（比如：只写一个登录框的校验，或者只写一个数据库连接模块）。
*   **目的：** 逼自己去面对那个黑乎乎的、什么都没有的编辑器。最难的往往是“第一行代码怎么写”，这种练习能帮你克服对空白文档的恐惧。

### 3. “先污染，后治理”（不要追求完美）
“眼高”的人往往有完美主义倾向，想一步到位写出和源码一样优雅的代码，写不出来就会卡住。
*   **做法：** 允许自己写出“垃圾代码”。先用最笨、最冗余的方法把功能跑通。
    *   代码重复？没关系。
    *   变量命名很烂？没关系。
    *   **只要它跑起来了，你就在“手”的层面赢了。**
*   **下一步：** 跑通之后，再去对比源码，看别人是怎么优化的。这时候的对比，你会印象极其深刻。

### 4. 强制 Debug 练习
“懂了”往往是忽略了细节，而 Debug 会逼你看清每一个细节。
*   **做法：** 故意把复刻好的代码改错一个地方，或者删掉一行，看看程序会报什么错。
*   **目的：** 了解代码运行的边界。很多时候你觉得自己懂了，其实你只是懂了“正确路径”，而真正的编程高手，懂的是“所有错误的坑在哪”。

### 5. “小量多次”的迁移式创新
不要 100% 复刻。
*   **做法：** 如果源码是做一个“待办事项清单（Todo List）”，你复刻的时候强迫自己改成做一个“书单记录卡”。
    *   底层逻辑是一样的（增删改查），但你换了字段名、换了 UI 样式。
*   **目的：** 这种微小的变动会切断你对源码的“视觉依赖”，强迫大脑进行二次思考。

### 6. 认清“记忆”与“搜索”的边界
不要因为记不住某个函数怎么写而感到挫败。
*   **标准：** 
    *   **不需要记：** `Array.prototype.splice()` 里面三个参数的具体顺序（记不住就查）。
    *   **必须知道：** 我现在需要删除数组中间的一个元素（这是逻辑）。
*   **解决：** 只要你知道“这个功能可以用某种方式实现”，然后能通过文档或 Google 快速查到写法，这就叫“会了”。

### 总结建议：
解决“眼高手低”最灵的药方只有四个字：**“折腾自己”**。

不要顺着舒服的路径（看视频、看源码、Copy）走，要逆着走——**先关掉参考资料，让自己痛苦地卡在那里，抓耳挠腮地想 20 分钟，实在不行了再去瞄一眼源码。**

那一眼，胜过你机械地抄写一百遍。
